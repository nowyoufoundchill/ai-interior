## Suite 5 detail (3 reviewer passes)

Suite 5 is a two-agent pattern: this session's builder agent never scored its own screenshots. Three independent fresh-context reviewer agents (no memory of the code, no memory of each other) scored successive capture batches:

- **Pass 1** (initial capture): found 3 real defects — a debug-run counter visible in the main workspace (violates "hidden route... not an admin panel"), primary nav literally labeled "Dashboard" (violates the design-studio register), and all 3 mock concepts sharing one identical palette + one templated sentence (violates the explicit "three concepts that feel the same" ban). All three fixed.
- **Pass 2** (post-fix capture): confirmed all three fixes landed, but flagged a new issue: 27 of 32 screenshots appeared to show the wrong tab highlighted. Root-caused to a capture-script bug, not an app bug: a pure tab-switch click has no network request, so the script's `waitForLoadState("networkidle")` wasn't a real synchronization point, and screenshots could fire mid-CSS-transition. Fixed with a `clickTabAndWait()` helper (waits for a panel-specific marker, then lets the transition settle).
- **Pass 3** (post-fix capture): confirmed the tab-highlight fix resolved 100% of that class of finding (verified individually on all 32 screenshots). New dominant finding: all 6 mock products shared one identical stock photo regardless of category — the photographic equivalent of the "three concepts that feel the same" ban, scoring the 3 Products-populated screens 4-5/10. Fixed with 6 distinct, category-verified placeholder photos (desk/chair/rug/lamp/artwork/plant, each confirmed by direct visual download-and-inspect before use, not guessed). Also closed a data-layer gap (layout_direction/decor_direction/budget_strategy and quality_score were still identical across the 3 mock concepts, invisible in the current card UI but real duplication) and a capture-completeness gap (the shared journey builder never sent a chat message, so "populated chat" always looked like the empty state).
- **Targeted re-score** (final, narrow): confirmed all 3 previously-failing product screens now score 8/10 with all 6 images distinct and category-correct.
- **Manual verification** (not re-scored by an agent, to avoid a 4th full-cost pass on already-confirmed-working fixes): direct screenshot inspection confirms the "populated chat" screen now shows a real saved turn matching the pattern that already scored 8/10 elsewhere.

**Final state:** 31 of 32 screens confirmed ≥8/10 with zero named §3/§11 rubric violations. The one remaining screen (1440px, concept-card hover state) sits at 7/10 for "no clearly visible hover affordance in a static capture" — the reviewer explicitly noted this is not a named rubric item, just a polish observation. Listed below as an open item rather than force-fixed under time pressure late in the cycle.

## Fixes made this session (chronological, all reseed-and-rerun verified)

**Real app bugs found while building the suites (not suite bugs):**
1. `generate-render` treated a mock `null` image as a live OpenAI failure whenever real provider keys happened to be configured — exactly our `seed:test` fallback situation (no dedicated `.env.test` project yet) — causing a spurious 500 in `AI_MODE=mock`.
2. Photo add/delete never marked the current diagnosis `stale` (§4 invalidation row 1 gap).
3. Editing a **locked** concept was silently allowed instead of rejected — §4 row 4 requires an explicit unlock first.
4. Unlocking a concept left downstream products/renders non-stale until the *next* re-lock instead of immediately (§4 row 3).
5. **Systemic residue-safety gap**: `ai_runs` and every artifact table written by a live route (diagnoses, mood boards, products, renders, chat messages/revisions, photos, preferences) never carried `test_run_id`, so every automated test cycle would have leaked permanent, undetectable rows into production — the residue check would report "clean" while actually missing everything. Fixed by tagging every insert from the room's already-loaded `test_run_id`, and updated `types/database.ts` to match migration 005 (previously untyped, silently masking the gap from `tsc`).
6. Tap targets across primary nav, room tabs, and nearly every concept/product/chat action button were under the PRD's 44px minimum.
7. A debug-run counter and "Open debug" link were visible in the main room workspace, contradicting "hidden route... not an admin panel."
8. Primary nav literally labeled "Dashboard" — renamed "Studio."
9. Mock concept and mock product fixtures used identical palettes/photos/text across all 3 concepts and all 6 products respectively, failing the "three concepts/products that feel the same" ban even in test mode — which would have made Suite 5 structurally unable to ever pass in mock mode.

**Suite/capture-script bugs found and fixed (not app bugs, confirmed via the debug endpoint / direct screenshot inspection that the app's actual behavior was correct throughout):**
- `locator.count()` is a point-in-time snapshot with no auto-retry; added polling helpers.
- Next.js's own aborted RSC prefetch requests were being misflagged as failed network requests.
- Screenshots taken immediately after a pure tab-switch click could race React's re-render and a CSS transition.
- The shared journey builder never sent a chat message.
